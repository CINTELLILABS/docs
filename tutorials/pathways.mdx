Pathways let you build structured dialogue flows using connected steps instead of relying only on a single prompt. Each step represents a stage in the conversation, and you can decide how the AI moves from one step to another based on user responses.

Pathways support different types of steps to control how a call progresses. The most commonly used nodes are the **Default Node**, which generates a response based on the node's description, and the **Webhook Node**, which connects to external APIs in real time. Additional nodes can route conversations, send SMS, transfer calls, and more.

# Key terminologies

<img
  style={{ borderRadius:"0.5rem" }}
  src="/tutorials/tutorials-assets/basic_nodes.png"
/>

**Nodes** are the building blocks of a pathway. Each node represents a step in the conversation where the AI performs an action—like asking a question, calling a webhook, or transferring the call.

**Edges** (also called pathways) are the connections between nodes. They determine which node the agent moves to next based on the user's response or other conditions.

**Routing** refers to how the AI navigates between nodes. The AI evaluates the user's response against available edge labels and picks the best match.

**Loop conditions** control when the agent can exit a node. The agent stays on the node until the condition is satisfied.

**Variable extraction** pulls structured information from the conversation into named variables that can be used in webhooks, prompts, and subsequent nodes.

**Global nodes** can be triggered from any point in the pathway. They're used for handling interruptions like FAQs or requests to speak to a human.

### Dialogue prompt

> **Note:** Enable **Static Text** to set a fixed response the agent always says.

## Transfer Call Node

The Transfer Node routes a live call from the AI to a human agent or an external phone number.

### Examples of when to use the Transfer Node

- The user requests to speak with a human
- The issue requires manual handling
- A high-value lead needs a sales representative
- The AI is unable to resolve the request

There are two types of transfer calls:

**Cold Transfer (Standard Transfer)** — The user is immediately transferred to another phone number without additional context being shared beforehand.

**Warm Transfer (Enterprise Feature)** — The AI initiates a separate call to brief the human agent on the customer's situation before connecting the user to the agent.

Learn more about the [Warm Transfer Enterprise Feature](https://docs.bland.ai/tutorials/warm-transfer)

Reference variables in prompts using double curly braces: `{{variable_name}}`.

## Built-in Variables

Built-in Variables are automatically available, no setup needed. These variables are extracted _during_ the call from the conversation itself. In pathway nodes (like webhook or scheduling nodes), you define variable names, types, and descriptions, and the AI extracts values from what was discussed.

Here are some examples of Built-in Variables:

- `{{phone_number}}` — Other party's number
- `{{timezone}}` — Timezone of the other party
- `{{country}}` — Country code
- `{{state}}` — State abbreviation
- `{{city}}` — Full city name
- `{{zip}}` — Zip code
- `{{call_id}}` — Unique call ID
- `{{now}}` — Current time in call's timezone
- `{{now_utc}}` — Current time in UTC
- `{{from}}` — Outbound number in E.164 format
- `{{to}}` — Inbound number in E.164 format
- `{{short_from}} / {{short_to}}` — Numbers without country code
- `{{lastUserMessage}}` — Last user response
- `{{prevNodePrompt}}` — Previous node's prompt

## Call into Variables (request_data)

Call into variables are variables you pass into a call at the time of sending it via the `request_data` field in the API request. These are not built-in. They would be custom variables you pass via request_data or extract during conversation.

They're accessible in prompts and pathways using `{{variable_name}}` syntax.

Some examples are:

- `{{party_size}}`
- `{{reservation_date}}`
- `{{reservation_time}}`

# Other Node Terminology

Within most Bland nodes, there are 3 main LLM functions that control the flow of the conversation:

- [Dialogue Generation](#dialogue-prompt)
- [Loop Conditions](#loop-condition)
- [Variable Extractions](#variable-extraction-prompts)

## Dialogue Prompt

The Dialogue Prompt sets the main goal for what your agent will say at that point in the conversation. Think of it as the script or instruction that guides your agent's response during a specific step of the call.

## Loop Condition

- Controls when the agent can exit the current node.
- The agent remains on the node until the condition is satisfied.
- Includes a likelihood score (e.g., 7/10) evaluated each turn.

**Example:**\
"Wait until the user provides a valid time and the agent confirms it."

## Variable Extraction Prompts

Pulls structured information from the conversation into named variables.

These variables can be used in:

- Webhook request bodies
- Prompts
- Subsequent nodes

Syntax for use: `{{variable_name}}`

## Global Node

- A node configured with: `isGlobal: true`
- And a `globalLabel` (e.g., "user asks a question").
- Can be triggered from any point in the pathway.
- Used for handling interruptions such as:
  - FAQs
  - Off-topic questions
  - Requests to speak to a human

## Unit Tests

Validate dialogue prompts, variable extraction, and loop logic on individual nodes using historical call data. You can generate user input permutations to stress-test prompts, and pin calls to auto-run with every test. This is part of the Standards framework for regression testing.

## Note Tag / Tag Name / Tag Color

Tags are metadata labels assigned to nodes or calls.

- Tag Name → The label text
- Tag Color → Visual identifier

Used for:

- Organizing calls
- Filtering call logs
- Categorizing pathways

Example filter:

- "Includes tag"
- "Does not include tag"

## Temperature

Controls the randomness/creativity of the model's responses at this node.

- Lower = More deterministic
- Higher = More creative

## Dynamic Interruption Threshold

Controls the sensitivity to user interruptions at this specific node. Adjusts how easily the user can interrupt the agent mid-speech.

## Skip User's Response

When toggled on, the agent proceeds to the next node without waiting for the user to respond.

## Block Interruptions

Prevents the user from interrupting the agent while it's speaking on this node. The agent will finish its full response before listening.

## Disable Repeating on Silence

When enabled, the agent will not repeat itself if the user is silent. Normally the agent may re-prompt after silence.

## Condition overrides global pathway

When enabled, this node's edge/routing conditions take priority over any global node conditions. Prevents global nodes from intercepting when this node's specific conditions should apply.

## Disable Recording

Stops call recording at this node for PCI compliance. Recording automatically resumes when the conversation exits the node.

## Disable Logging

Stops transcript logging at this node for sensitive data handling. Like recording, it resumes automatically after leaving the node.

# Standards

Standards let you build better agents, faster, by defining scenarios, backtesting prompts against them, and grading how those prompts perform.

## Pathway (Edges)

Pathways are the connectors between nodes and determine which next node the agent can move to, usually determined by how the user responds to your assistant.

## Routing

Refers to how the AI agent navigates between nodes in a Conversational Pathway.

### LLM Based Routing

The LLM evaluates the user's response against the available edge labels and picks the best match.

For example, from a "Start" node, you might have edges labeled:

- "User wants to reserve"
- "Live Agent Requested"

The agent chooses based on the context of the user response.

Handles branching logic and conditional navigation within pathways, determined by a variable which could be a request_data or call-into variable.

# Testing the Pathway Agent

Test your pathway agent by clicking **Test** in the pathway toolbar. There are three testing modes: web-based chat (text or voice), sending a real call to a phone number, and per-node unit testing.

<img style={{ borderRadius: "0.5rem" }} src="/tutorials/tutorials-assets/testing-hero.png" />

## Test Configuration

Each testing mode shares a common set of configuration options:

- **Load configuration** — Restore a previously saved test setup.
- **Start node** — Choose which node the conversation begins at. Defaults to the start node. Useful for testing nodes deeper in the pathway without replaying the full conversation.
- **Version** — Select which pathway version to test against (e.g., the current draft or a published version).
- **Request data** — Add key-value pairs representing context the agent should know (e.g., a caller's name or account ID). The agent sees both keys and values, and they're accessible via `{{variable_name}}` syntax in your prompts.
- **Use Candidate Model** — Test your pathway against a candidate model before promoting it.
- **Save configuration** — Save your current setup as a named configuration for reuse.

## Text Chat

The most common way to test. Type messages in the input field and the agent responds as it would on a live call. Each message displays a timestamp. As the conversation progresses, you can track which node the agent moves to next — when the route and loop condition steps resolve, the log shows the destination node for the following message.

## Voice Chat

Switch to the **Voice** tab to test your agent with speech. Voice-specific configuration includes:

- **Voice** — Select the voice the agent will use.
- **Language** — Choose the conversation language (e.g., English via Babel).
- **Block interruptions** — When enabled, prevents you from interrupting the agent while it's speaking.
- **Pronunciation Guide** — Add word-level pronunciation overrides for terms the agent may mispronounce (e.g., brand names, medical terms).

## Send Call

Click **Send call** to trigger a real phone call using your pathway configuration. The call behaves identically to a production call, and logs appear in the testing panel in real time.

## Branching

In any chat mode, you can branch a conversation from any user message to explore how the agent handles a different input at that point.

To create a branch, click the edit icon on a user message, change the text, and submit it. The agent generates new responses based on the updated message, and the conversation continues from that point forward. The original branch is preserved — you can swap between branches using the arrow controls that appear on the branched message (e.g., `1/2`).

This is useful for quickly testing how the agent responds to variations of a user message without restarting the conversation from the beginning.

<img style={{ borderRadius: "0.5rem" }} src="/tutorials/tutorials-assets/testing-branch.png" />

## Expanded Call Logs

Click **Expand all logs** to see detailed per-turn decision data for any test conversation. Each turn shows the following sections when relevant:

**Variable extraction** — Displays the extraction type and results. When variables are successfully extracted, their names and values appear (e.g., `party_size: 8`, `appointment_time: 9:30 AM`). The full list of available variables from previous nodes is also shown, along with the node each variable originated from.

**Loop condition** — Shows whether the loop condition was evaluated and its result. Includes the full loop condition prompt and a likelihood score (e.g., `Likely 7/10`). This is the only prompt shown directly in the expanded logs — node prompts and the global prompt are accessible from the node editor panel and the global prompt window respectively. When the condition is not yet met, the agent remains on the current node.

**Route** — Displays which outgoing pathway the agent chose and the other available options it could have taken.

**Interruption** — When a user interrupts the agent, the log shows the interruption event, including which node the agent returned to.

**Webhook** — For nodes that trigger webhooks, the log shows the URL, HTTP response status, response time, and the full response payload.

## Unit Testing

<img style={{ borderRadius: "0.5rem" }} src="/tutorials/tutorials-assets/testing-unit-tests.png" />

Unit testing lets you create reusable test cases for individual nodes using historical call data. This is useful for validating that prompt changes don't break existing behavior.

To set up a unit test:

1. **Select a node** — Choose the node you want to test from the dropdown.
2. **Select a Call ID** — Pick a historical call to use as conversation context.
3. **Select a user message** — Choose a specific user message from that call to replay against the node.
4. **Replacement message** *(optional)* — Override the original user message with custom text to test how the agent handles different inputs.
5. **LLM grading goal** *(optional)* — Describe what the agent's response should accomplish. This is used as the evaluation criteria for the test.

**Simulated variations** — Toggle this on to generate multiple variants (default: 5) of the user message with different communication styles. This stress-tests your prompt against a range of phrasings to ensure robustness.

Click **Run test** to execute. Results are evaluated using a **prompt-based** test — an LLM evaluates whether the agent's response satisfies the grading goal (or the node prompt if no custom goal is provided) and returns a Pass/Fail result along with a rationale.

When simulated variations are enabled, each variant shows its own response, extracted variables, and individual test results. You can expand any variant to see the full agent response and the grading rationale. A summary badge (e.g., `1/1 failed`) appears at the top of the test for quick pass/fail visibility.

# Variables

Reference variables in your pathway prompts using double curly braces: `{{variable_name}}`.

## Passing Variables into a Call

Pass variables into a call by including key-value pairs in the `request_data` field when sending a call. These become accessible in any node prompt using `{{variable_name}}` syntax.

## Built-in Variables

The following variables are available at every node without any configuration:

- `{{lastUserMessage}}` — The last response from the user.
- `{{prevNodePrompt}}` — The prompt from the previous node.
- `{{now_utc}}` — The current time in UTC.
- `{{from}}` — The phone number the call is from.
- `{{to}}` — The phone number the call is to.
- `{{call_id}}` — The unique identifier for the call.

## Extracting Variables from the Conversation

At each node, you can extract variables from the conversation so far using the **Extract Variables from Call Info** section. For each variable, specify:

- **Name** — The variable name you'll reference later (e.g., `party_size`).
- **Type** — `string`, `integer`, or `boolean`.
- **Description** — What information the variable should capture. The more specific this is — including expected formats and examples — the more accurately the agent will extract the value.

Extracted variables become available in subsequent nodes using the same `{{variable_name}}` syntax.

**Latency note:** Enabling variable extraction introduces slight latency, since the agent must extract variables from the conversation before generating dialogue for the next node.

**Webhook nodes:** Variable extraction runs *before* the webhook executes, so you can reference extracted variables in the webhook's request data. Variables returned in the webhook's `response_data` are also available in the dialogue generated after the webhook completes.
